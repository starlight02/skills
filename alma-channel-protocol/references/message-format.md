# Message Format & Source Routing Reference

This document covers the message text format conventions used by Alma's channel bridges,
the `source` field routing table, and the `channel_mappings` database schema.

## Message Text Format

Every bridge prefixes user messages with sender identification for the AI's context.

### Private Format

```
[msg:message_id] message text
```

### Group Format

```
[From: DisplayName (@username) [id:platform_id] [msg:message_id]] message text
```

This is the format used by Alma's bundled Telegram and Discord bridges. Do not split the
sender header and `[msg:message_id]` into separate lines for group messages.

### With Reply/Quote

```
[From: DisplayName (@username) [id:platform_id] [msg:message_id]] [Replying to Sender's message: "quoted text"]
message text
```

### Platform-Specific Examples

**Telegram (private)**:
```
[msg:42] Hello, how are you?
```

**Telegram (group)**:
```
[From: Alice (@alice_wonder) [id:123456789] [msg:789]] Does anyone know the answer?
```

**Telegram (with reply)**:
```
[From: Bob (@bob_dev) [id:987654321] [msg:790]] [Replying to Alice's message: "Does anyone know the answer?"]
Yes, the answer is 42.
```

**Discord**:
```
[From: Alice (@alice) [id:1234567890123456789] [msg:1234567890123456790]] Hello!
```

**QQ/OneBot (group, using Telegram spoof)**:
```
[From: 萌依 [id:123456789] [msg:12345]] 大家好
```

**QQ/OneBot (private)**:
```
[msg:67890] 你好
```

**QQ/OneBot (with reply)**:
```
[From: 萌依 [id:123456789] [msg:12346]] [Replying to Alice's message: "帮我查一下天气"]
好的，我来查一下
```

### Format Field Reference

| Field | Source | Required | Notes |
|-------|--------|----------|-------|
| `DisplayName` | Platform sender name | Yes | Telegram: first_name; Discord: display name; QQ: group card or nickname |
| `@username` | Platform handle | No | Omit if platform doesn't provide usernames (QQ) |
| `[id:N]` | Platform user ID | Group only for Telegram-style text | Stable identifier (nicknames change) |
| `[msg:N]` | Platform message_id | Yes | Real message ID from the event (not a counter) |
| `[Replying to...]` | Quoted message context | No | Generated by `buildReplyContext()` — see reply-quoting.md |

### `[msg:N]` Rules

- **Must be the real platform message_id** from the incoming event
- For Telegram: `message.message_id` (integer, sequential per chat)
- For Discord: `message.id` (Snowflake string)
- For QQ/OneBot: `event.message_id` (integer)
- **Never hardcode to 0** — this breaks message referencing in group history
- For group messages, `[msg:N]` belongs inside the `[From: ...]` header. For private
  Telegram-style messages, `[msg:N]` is the standalone prefix.
- Used by Alma in group history logs for cross-referencing

## The `source` Field — Complete Reference

### Source Values and Effects

| Source | Channel | Group? | History Stripping | System Prompt Added |
|--------|---------|--------|-------------------|-------------------|
| `"telegram"` | Telegram | No | Yes | Telegram formatting, file/sticker/voice |
| `"telegram-group"` | Telegram | Yes | Yes | All above + group rules (privacy, observation, frequency) |
| `"discord"` | Discord | N/A | Yes | Discord markdown, channel directory, group behavior |
| `"feishu"` | Feishu | No | Yes | Feishu-specific rules |
| `"lark"` | Lark | No | Yes | Same as Feishu (international variant) |
| `"weixin"` | WeChat | No | **No** | WeChat file/voice, plain text formatting |
| `"heartbeat"` | Internal | No | No | Heartbeat agent turn |
| `"cron"` | Internal | No | No | Scheduled task agent turn |
| `"system-commitment-followup"` | Internal | No | No | Auto-injected promise follow-up |

### Custom Bridge Source Selection

| Your Bridge Type | Use Source | Rationale |
|-----------------|------------|-----------|
| Group chat (any platform) | `"telegram-group"` | Gets group rules, privacy firewall, history stripping |
| Private/DM chat | `"telegram"` | Gets private chat treatment, history stripping |
| Platform with no group distinction | `"telegram"` | Simplest option with full processing |

### What Each Source Gives You

**`"telegram-group"` (recommended for group bridges)**:
- Group chat system prompt: privacy firewall, people observation, frequency control
- History stripping of stale ephemeral content
- Telegram-specific formatting rules
- Group interaction guidelines (@mentions, observation behavior)

**`"telegram"` (recommended for private bridges)**:
- Private chat treatment
- History stripping
- Telegram formatting rules
- File/sticker/voice sending instructions

**`"weixin"` (WeChat-specific)**:
- WeChat-specific rules only
- **No history stripping** — stale profiles accumulate in conversation history

## Group Chat Detection

| Channel | Detection Method | Source Used |
|---------|-----------------|-------------|
| Telegram | `chatId < 0` (negative = group) | `"telegram-group"` |
| Discord | Always in guild; DMs handled separately | `"discord"` (same for both) |
| Feishu | `isGroupChat` boolean from handler | `"feishu"` (same for both) |
| Weixin | No explicit distinction | `"weixin"` (same for both) |
| QQ/OneBot | `message_type == "group"` | `"telegram-group"` or `"telegram"` |

## `channel_mappings` Table

```sql
CREATE TABLE channel_mappings (
    id TEXT PRIMARY KEY,
    platform TEXT NOT NULL,
    external_chat_id TEXT NOT NULL,
    external_user_id TEXT NOT NULL,
    thread_id TEXT NOT NULL REFERENCES chat_threads(id) ON DELETE CASCADE,
    is_active INTEGER DEFAULT 1,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

### Lookup Key

`(platform, external_chat_id, external_user_id)` — uniquely identifies a conversation.

### Platform-Specific ID Conventions

| Platform | Private Chat `external_user_id` | Group Chat `external_user_id` |
|----------|-------------------------------|------------------------------|
| Telegram | Sender's Telegram ID (string) | `"group"` |
| Discord | Sender's Discord user ID | Sender's Discord user ID |
| Feishu | Sender's Feishu user ID | `"group"` |
| Weixin | chat ID (= external_chat_id) | N/A |
| QQ/OneBot | Sender's QQ ID (string) | `"group"` |

### Outgoing Routing: `detectPlatformForChatId`

```javascript
detectPlatformForChatId(chatId) {
  const row = db.prepare(
    "SELECT platform FROM channel_mappings WHERE external_chat_id = ? AND is_active = 1 LIMIT 1"
  ).get(chatId);
  if (row?.platform === "discord") return "discord";
  if (row?.platform === "weixin") return "weixin";
  return "telegram";  // DEFAULT FALLBACK
}
```

**Implications**:
- `platform = "telegram"` works for any custom bridge — routing falls back to Telegram
- Only `"discord"` and `"weixin"` have explicit routing; everything else → Telegram
- Custom bridges should use `platform = "telegram"` for compatibility

### Creating Mappings for Custom Bridges

```sql
-- Group chat
INSERT INTO channel_mappings (id, platform, external_chat_id, external_user_id, thread_id, is_active, created_at, updated_at)
VALUES ('uuid', 'telegram', '<qq_group_id>', 'group', '<thread_uuid>', 1, datetime('now'), datetime('now'));

-- Private chat
INSERT INTO channel_mappings (id, platform, external_chat_id, external_user_id, thread_id, is_active, created_at, updated_at)
VALUES ('uuid', 'telegram', '<qq_user_id>', '<qq_user_id>', '<thread_uuid>', 1, datetime('now'), datetime('now'));
```

## Group Chat History Format

Alma's `alma group list/history/search/context` commands read plain log files from
`~/.config/alma/groups/<chatId>_<YYYY-MM-DD>.log`:

```text
[14:30:00] [msg:42] [萌依 [id:123456789]]: 大家好
[14:31:00] [msg:43] [Alma (BOT)]: 你好！有什么可以帮忙的吗？
[14:32:00] [msg:44] [Alice [id:987654321]]: [Replying to Alma's message: "你好！"] 帮我查一下天气
```

Format: `[HH:MM:SS] [msg:N] [DisplayName [id:userId]]: text`

- `[msg:N]` uses real platform message IDs
- `[Alma (BOT)]` marks the AI's own messages in persistent logs
- Reply context uses the same `[Replying to X's message: "..."]` pattern
- Recent in-memory entries are also included in `ephemeralContext` as a `RECENT CHAT HISTORY` block
- Active sends for custom platforms should use a bridge-owned endpoint/tool; Alma's `alma group send` is Telegram-specific
