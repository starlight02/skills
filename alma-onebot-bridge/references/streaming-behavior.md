# Streaming Behaviour, Stage Messaging, and Tool Calls

How the bridge consumes Alma's WS event stream and turns it into QQ messages. This is the trickiest behaviour to get right because Alma streams in mid-generation, fires intermediate `generation_completed` events around tool calls, and expects the bridge to emit progress messages without polluting the final reply.

## Lifecycle of One Generation

1. Bridge sends `generate_response` over the Alma WS.
2. Alma emits a stream of events. The bridge accumulates visible text from:
   - `message_delta` events with `text_append` and `partType: "text"` (or `"output_text"`)
   - `message_delta` events with `text-delta`
   - `part_add` events that add a `text` part
3. When a `part_add` introduces a **tool invocation** part, the bridge:
   - Sends the visible text accumulated so far as a **stage message** to QQ.
   - Records the visible prefix that was sent so the final reply can avoid resending it.
   - Keeps the generation pending. The `generation_completed` / `thread_generating { isGenerating: false }` pair that immediately follows a tool invocation is **not** the end — wait for the continuation after the tool returns.
4. While the tool runs, Alma may stream tool input over multiple `tool_input_append` events. If `chat.show_tool_calls` is true, the bridge waits until input parameters are available (or the tool leaves `input-streaming`), then emits a separate progress message:
   ```
   正在调用工具：WebSearch
   参数：{"query":"..."}
   ```
   These status messages are progress notifications. They **never** become final-reply prefixes, and they do not change paragraph-splitting semantics.
5. When generation truly completes, the bridge sends the **remaining** text after the recorded stage prefix as the final message.

## Idle Timeout Resets on Progress

`alma.timeout` is treated as an idle timeout, not a wall-clock timeout. Every emitted stage/tool-call progress event resets the wait window so a long-running tool call does not force a premature retry. The retry loop only fires if Alma genuinely stops streaming for `timeout` seconds.

## Think Block Stripping

Alma emits `<think>...</think>` blocks in some model outputs. Strip them from all visible text **unless** `chat.show_thinking` is enabled. When enabled, thinking blocks are surfaced as separate QQ messages.

## Reply Threading and `@`-Mention

QQ wants the reply reference attached to the first visible message in a bridge-owned turn so the conversation threads correctly on the user side. The bridge applies these rules:

- The first **successfully sent** visible stage / tool-call status / final message in a turn carries the outgoing reply reference and, for group messages, the `@<sender>` mention.
- Every later visible message in that turn is plain — no reply reference, no `@`-mention.
- If a stage or tool-call message already consumed the reply, the final message is plain.

This differs from Alma's built-in Telegram bridge, which always attaches the reply to the final `sendReply`. The deviation is deliberate: stage messages in QQ are visible to the user, while in Telegram they are deferred.

## Dedup Against the Forwarding Loop

The Alma WS reader broadcasts `message_updated` events; the OneBot writer forwards them to QQ when the message is generated by the GUI rather than by the bridge itself. To avoid echoing bridge-owned replies back to QQ, the bridge keeps a per-thread dedup buffer.

Dedup compares **normalized visible assistant text**:

- Strip HTML breaks (`<br/>` etc.).
- Collapse whitespace.
- Strip `<think>...</think>` blocks.
- Strip system reminder boilerplate.

Then a bidirectional prefix match (`sent.starts_with(prefix) || prefix.starts_with(sent_prefix)`) decides if the candidate matches a recently sent reply. The buffer keeps the last ~20 entries per thread. This keeps distinct messages with a shared long prefix from being mistaken for duplicates, while still catching the bridge's own replies arriving back through `message_updated`.

## `chat.segmented_replies` vs Stage Messaging

These are independent settings. Operators sometimes conflate them; do not.

| Setting | What it does | When it applies |
|---|---|---|
| Stage messaging | Sends visible text accumulated before a tool boundary as a separate message. Always on; matches Alma's built-in Telegram behaviour. | At tool-call boundaries during a generation. |
| `chat.segmented_replies` | When true, splits each stage/final text by paragraphs (`\n\n`) before the QQ 4500-character chunker runs. | For presentation only; off by default to keep replies as one QQ message. |

QQ's ~4500-character limit always triggers chunking regardless of these settings; the chunker prefers line breaks at the boundary.

## Per-Thread Generation Guards

A `HashMap<String, Arc<smol::lock::Mutex<()>>>` keyed by `thread_id` serializes concurrent `generate()` calls for the same Alma thread. Without it, a second concurrent call would overwrite the first call's pending oneshot sender, corrupting the API correlation table.

Pattern:

```rust
let guard = {
    let mut guards = self.guards.lock().await;
    guards.entry(thread_id.to_string())
        .or_insert_with(|| Arc::new(smol::lock::Mutex::new(())))
        .clone()
};
let _hold = guard.lock().await;
// safe to send generate_response and await the response
```

The guard is per-thread, not global. Different threads can stream concurrently.

## Group History as Context

When `chat.group_history_size > 0`, the bridge ships the last `group_history_size` non-bot messages from the in-memory ring buffer into `ephemeralContext`. This is independent of the `~/.config/alma/groups/<id>_<date>.log` file, which is written for `alma group history/search/context` to consume.

The bridge writes the on-disk log even when `respond_to_group_messages` is false, as long as `listen_group_messages` is true. This keeps Alma's CLI tools useful even when the bridge is muted in groups.

## Active Sends From Alma Tools

Alma's built-in `alma group send` is Telegram-only. For QQ, expose the bridge's HTTP command endpoints and inject usage instructions into `ephemeralContext` (or as part of the group README) so Alma tools call the bridge directly:

```bash
curl -s -X POST http://127.0.0.1:8090/qq/group/<group_id>/send \
  -H 'Content-Type: application/json' \
  -d '{"message":"hello"}'

curl -s -X POST http://127.0.0.1:8090/qq/private/<user_id>/send \
  -H 'Content-Type: application/json' \
  -d '{"message":"hello"}'
```

Loopback callers bypass token requirements. Non-loopback callers must present `Authorization: Bearer <onebot.access_token>` or `?token=...`.

The optional `reply_to_id` field on the JSON body attaches an outgoing reply reference to the **first** chunk only. `at_user_id` adds an `@` mention to the same first chunk. Both are useful when Alma is replying to a specific message in the group rather than starting a new conversation.

## QQ Face Expression Decoding

QQ face IDs are not human-readable. The bridge maps the most common "super expression" IDs to Chinese names via `face_map.rs`. Unknown IDs fall back to `[emoji:face_<id>]`. The map covers ~180 IDs sourced from go-onebot-agent / NapCat. When you add new IDs:

- Keep ID-to-name and name-to-ID in sync (the reverse lookup is built on first access).
- Test that duplicate face names keep the first known ID (e.g. `264` and `319` both map to `捂脸`, but `face_id("捂脸")` returns `264`).
